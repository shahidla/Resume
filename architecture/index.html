Case study 01

S/4HANA modernisation and clean-core readiness

Shahid supported clean-core and cloud-readiness initiatives across Commonwealth Bank, Roy Hill, Department of Corrections New Zealand, Woolworths, and Services Australia. The work focused on reducing legacy risk, preparing custom code for S/4HANA, and aligning development patterns with cloud-compliant architecture.

Architecture approach

  • Assessed custom object usage with ABAP Call Monitor, SCMON, and SUSG.
  • Used SAP Custom Code Migration App, SAP Readiness Check, ATC, and Fiori Custom Code Migration.
  • Applied automated Quick Fix where suitable and manual remediation where business logic required architectural judgement.
  • Advocated tiered extensibility: Tier 1 cloud extensibility, Tier 2 cloud API enablement, and Tier 3 classic ABAP extensions.
  • Refactored custom developments using CDS Views, AMDP, SQL Monitor, and SQL Trace for performance and clean-core alignment.

Representative outcomes

Delivered modernised artefacts, retired or remediated legacy custom code, prepared systems for upgrade and transformation programs, and helped teams understand practical S/4HANA readiness decisions rather than treating clean core as only a compliance checklist.

Case study 02

AI-assisted SAP RAP execution with an MCP server

As part of AI-assisted SAP automation exploration, Shahid built a Node.js-based MCP server that exposes SAP RAP OData services as controlled tools. The goal is to allow AI-assisted execution of business tasks without direct SAP GUI interaction.

Architecture approach

  • Used Node.js as the controlled tool server layer.
  • Exposed SAP RAP OData services through explicit tool contracts.
  • Separated AI orchestration from direct SAP interaction to reduce operational risk.
  • Designed the pattern so business operations can be mediated, audited, and constrained.

Why it matters

This positions SAP automation beyond chatbot demos. The architectural value is in controlled execution: business services remain in SAP, while AI receives a narrow, governed interface for task execution.

Case study 03

High-volume processing and HANA code pushdown

Shahid has repeatedly worked on performance-critical SAP programs where data volume and runtime matter. Examples include Woolworths SAP CAR UDF developments processing more than 40 billion records and Commonwealth Bank data scrambling reduced from 240 hours to 6 hours.

Architecture approach

  • Used ABAP on HANA, CDS, AMDP, RAP, SQL optimisation, and code pushdown.
  • Analysed runtime with Code Inspector, SQL Trace, AMDP Trace, PlanViz, and SQL Monitor.
  • Designed processing around partitioned tables, database-side execution, and scalable batch patterns.
  • Refactored legacy logic to reduce application-server work and exploit HANA execution plans.

Representative outcomes

Delivered measurable runtime improvement, enabled stable non-production system copy processes, and supported machine-learning reporting and high-volume forecast data calculations in retail environments.
